# proposal-writing

사업 수주를 위한 제안서 작성 보조. AI 솔루션 기업의 사업 팀장 관점에서 작성됨. 외부 산출물이므로 work-style의 "반말/축약형" 규칙은 **적용되지 않음** — 본문은 개조식, 대화는 work-style 유지.

## 언제 활성화되나

- "제안서/제안 발표자료/과제 신청서/사업계획서/RFP 응답" 키워드
- 사용자가 RFP, 과거 제안서, 명세서, 제안서 초안을 공유할 때
- "이 섹션 다듬어줘", "초안 써줘", "개조식으로 바꿔줘" 류 요청

## Claude의 역할 (Jungwoo의 요청 범위)

작업은 보통 셋 중 하나:
1. **본문 초안 작성**: 주어진 구조/목차에 따라 슬라이드·섹션 본문 채우기
2. **자료 재구성**: 과거 제안서·명세서·기술문서를 이번 과제에 맞게 재가공
3. **초안 다듬기**: Jungwoo가 쓴 초안의 표현·논리 강화

**구조/목차 설계는 Jungwoo가 한다.** Claude가 멋대로 목차를 새로 짜지 않음. 단, "구조도 같이 잡아줘" 요청 시 예외.

## 본문 작성 규칙 (가장 중요)

### 개조식 only

- **명사형 종결**: "~함", "~임", "~됨", "~적용", "~구축", "~확보".
- 줄글 금지. "저희는 X를 통해 Y합니다" → "X 기반 Y 실현".
- bullet (`-`) 또는 짧은 문단(2–3줄). 한 bullet = 한 메시지.
- 주술 호응 무리하지 말 것. 어색하면 끊는다.

**예시**:
- 좋음: `- 발주처 ○○도메인 5년간 구축 경험 보유 (사례 3건, 후술)`
- 나쁨: `- 저희는 ○○도메인에서 5년간 구축 경험을 보유하고 있으며, 이는 본 과제 수행에 큰 강점이 됩니다.`

### 한국어 어기 일관성

- 한 문서 안에서 "~함" / "~임" 중 하나의 종결 패턴을 일관되게.
- 섞이면 어색하므로, 작성 시작 전 Jungwoo에게 1회 확인하거나 기존 문서의 패턴을 따름.

### PPT vs Word

- **본문 톤은 동일** (개조식). 차이는 분량.
- PPT: 1슬라이드 1메시지 원칙. 키워드+근거 1–2줄. 표·다이어그램·로고 제안은 사용자에게 위임 (별도 ppt-design-system skill 예정).
- Word/HWP: 같은 개조식이되 근거·디테일·예시 더 풍부히.

## 절대 쓰지 말 것 (금기 표현)

평가자가 가장 싫어하는 표현. 작성 시 자가검열, 제출 전 자동 스캔 대상:

| 금기 | 이유 | 대안 |
|---|---|---|
| "최선을 다하겠습니다" | 공허, 약속 아님 | 구체 일정·산출물·KPI 명시 |
| "혁신적인", "최고의", "세계 최초" (근거 없이) | 장식어 | 특허/논문/벤치마크 인용 |
| "정확도 95% 향상" (출처 없이) | 검증 불가 수치 | 기준선·측정조건·출처 명시 |
| "가능하도록 노력" | 모호 | "X를 함" 단언, 또는 조건 명시 |
| 경쟁사 직접 비방 | 평가 감점 | 자사 강점 단언으로 우회 |

## 4대 싸움터 (Win Themes)

모든 제안서가 이기고 싶어하는 네 영역. 섹션 본문 쓸 때 이 중 어떤 싸움터인지 인식하고 그에 맞는 무기를 씀:

1. **고객 도메인 이해/공감**: "발주처 X 페인포인트", "현장 Y 제약", "유사 기관 Z 사례 인지" — 발주처 언어를 그대로 인용.
2. **기술적 차별화**: 자체 엔진/특허/논문/모델. "우리만 할 수 있는 것"이 명확해야 함. 모호하면 차별화 아님.
3. **실행 능력/트랙레코드**: 유사 도메인 구축 사례, 인력 이력, 운영 경험. 숫자·고객명·기간으로 단언.
4. **ROI/비용 효율**: 투자 대비 효과, 운영 비용 절감, TCO. 수치는 가정·기준선 명시.

본문 한 단락이 위 넷 중 어느 것에도 안 걸리면 분량 낭비. 빼거나 다른 싸움터로 재구성.

## 작업 흐름

1. **자료 수집**: RFP, 과거 제안서, 명세서를 공유받음. 없으면 요청.
2. **회사·제품 정보**: `company-context` skill에서 자동 참조 (없으면 사용자에게 묻거나 자료 첨부 요청).
3. **하이레벨 초안 먼저**: 전체 목차에 대해 키 메시지·핵심 근거만. 디테일은 나중.
4. **챕터 단위 깊이 작업**: 사용자가 "X챕터 디테일 써줘" 하면 그때 본격 작성. 한 번에 다 쓰려 들지 않음.
5. **시간 압박 가정**: 80% 퀄리티로 빠르게가 default. "더 다듬어줘" 요청 시에만 시간 들임.

## 제출 전 자동 체크리스트

사용자가 "제출 전 체크해줘", "마지막 점검", "검토" 등 신호 시 다음 4개 자동 점검:

1. **금기 표현 스캔**: 위 표의 금기 항목 grep.
2. **RFP 요구사항 대조**: 원본 RFP 평가항목·의무 제출자료 리스트와 본문 교차 검증. 누락·불완전 항목 보고.
3. **용어 일관성**: 제품명, 조직명, 약어가 문서 전반에서 동일하게 쓰였는지. 첫 등장 시 풀어쓰고 약어 정의했는지.
4. **개조식 어기 일관성**: "~함/임/적용" 등 명사형 종결이 한 패턴으로 통일됐는지. 혼재 시 보고.

체크 결과는 **{발견 항목, 위치, 제안 수정}** 표로 정리.

## company-context skill과의 관계

- 회사 차별화·트랙레코드·인력·고객사 등은 별도 `company-context` skill 영역.
- 그 skill이 아직 없거나 정보가 부족하면 **사용자에게 1회 묻기**: "○○ 영역 차별화 포인트 자료 있어? 없으면 알려줘."
- 추측해서 채우지 않음 (work-style 금기와 일치).
